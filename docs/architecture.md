# 🏗️ Архитектура системы: Checkers vs Chess

Данный документ описывает внутреннее устройство проекта, разделение ответственности между компонентами и логику работы гибридного игрового движка.

## 1. Общий обзор (High-Level View)

Архитектура следует упрощенному паттерну **Model-View-Presenter (MVP)**. Основная логика игры вынесена в чистые Kotlin-классы, отделенные от Android-фреймворка. Это обеспечивает:
* Высокую скорость работы ИИ.
* Возможность глубокого Unit-тестирования (Bot Arena).
* Легкую модификацию правил без изменения UI.

## 2. Слой Модели (Data Model)

Модель представляет собой текущее состояние игрового мира.

| Класс | Описание |
| :--- | :--- |
| **`Board`** | Хранит состояние поля (`Array<Array<GamePiece?>>`). Управляет выполнением ходов и проверкой условий победы. |
| **`GamePiece`** | Абстрактный базовый класс. Хранит цвет (`PieceColor`) и логику клонирования для симуляций. |
| **`ChessPiece`** | Наследник `GamePiece`. Содержит тип фигуры (`ChessPieceType`), её стоимость и штрафы за потерю. |
| **`CheckerPiece`** | Наследник `GamePiece`. Хранит статус «Дамки» и логику движения шашек. |
| **`Move`** | DTO-класс, описывающий координаты хода и список захваченных фигур (`captured`). |

## 3. Слой Логики (Game Logic)

Разделение правил двух игр реализовано через специализированные валидаторы и системы ИИ.

### **`MoveValidator`**
Центральный узел правил. Отвечает за:
* Генерацию доступных ходов для любой фигуры.
* **Гибридный расчет:** расчет траекторий шахматных фигур против шашек и наоборот.
* Проверку безопасности Короля (`isKingInCheck`).

### **`Bot System`**
Система построена на интерфейсе `Bot`, что позволяет подставлять разные алгоритмы:
* **`PrimitiveBot`:** Случайный выбор хода.
* **`SmartChessBot`:** Реализация **Minimax** с Alpha-Beta отсечением.
    * **Глубина:** 4 полухода.
    * **Quiescence Search:** Дополнительный поиск для оценки позиций с незаконченными взятиями (предотвращает «зевки»).
    * **Heuristic Evaluation:** Сложная оценочная функция, учитывающая ценность фигур, их позицию и угрозы.

## 4. Жизненный цикл хода (Turn Execution Flow)

1. **Запрос хода:** Система ожидает ввод от игрока или вызывает `Bot.getMove()`.
2. **Симуляция (для ИИ):** Бот создает копии доски через `simulateMove()`, просчитывая дерево решений.
3. **Валидация:** `MoveValidator` подтверждает легальность выбранного действия.
4. **Выполнение:** `Board.makeMove()` обновляет массив фигур, удаляет захваченные объекты и проверяет условия превращения (пешка -> ферзь, шашка -> дамка).
5. **Штрафы:** UI-контроллер получает информацию о потерях и вычитает время из таймеров согласно `ChessPieceType.penalty`.

## 5. Система таймеров

Логика времени работает асинхронно:
* Каждый игрок имеет свой `CountDownTimer`.
* Время — это основной ресурс. Победа достигается не только матом или уничтожением фигур, но и доведением времени противника до нуля.
* Штрафы за потерю фигур или шах мгновенно корректируют значение миллисекунд в активных таймерах.
